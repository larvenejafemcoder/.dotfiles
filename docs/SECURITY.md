## Privilege Model

### Root / Sudo Usage

The script requests root privileges **only** for specific operations:

| Operation | Why root is needed | How it's obtained |
|-----------|-------------------|-------------------|
| Package install | Writing to `/usr/lib`, `/var`, etc. | `sudo_check()` → `sudo -v` → `$PKG_INSTALL` |
| Go installation | Writing to `/usr/local/go` | `sudo rm`, `sudo tar` |
| Docker install | Adding repos, modifying `/etc` | `curl | sh` (via sudo context) |
| PPA management | Adding apt sources | `sudo add-apt-repository` |
| Neovim PPA | Adding apt sources | `sudo add-apt-repository` |

### `sudo_check()` mechanism

```bash
sudo_check() {
    if ! sudo -n true 2>/dev/null; then
        setup_warn "Sudo needed: $*"
        sudo -v
    fi
}
```

- `sudo -n true` — non-interactive test: returns 0 if cached, 1 if prompt needed
- `sudo -v` — refreshes the credential cache (prompts the user once)
- The session caches for ~5 minutes (default `timestamp_timeout`)

> [!WARNING]
> The script intentionally **does not** run `sudo` on the entire process.
> Only individual commands are prefixed with `sudo`. This minimizes the
> blast radius of a bug in any single operation.

### What does NOT need root

- Dotfile symlink creation (`ln -sf` — user owns `$HOME`)
- Backup creation (`cp` — user's config directory)
- TUI interaction (dialog/whiptail — no privileges needed)
- Logging (user's config directory)
- `source` / `curl | sh` installers (nvm, rustup, starship)

## Download Verification

### Current state

The script uses `curl -fsSL` to download installers. Verification:

| Installer | Download | Integrity check |
|-----------|----------|----------------|
| nvm | `raw.githubusercontent.com` | None (HTTPS) |
| rustup | `sh.rustup.rs` | None (HTTPS) |
| Go | `go.dev/dl/go*.tar.gz` | None (HTTPS) |
| Starship | `starship.rs/install.sh` | None (HTTPS) |
| Docker | `get.docker.com` | None (HTTPS) |
| zoxide | `raw.githubusercontent.com` | None (HTTPS) |

All downloads use **HTTPS** (TLS 1.2+), which provides transport-layer
security (mitigates MITM on the wire). However, there is **no**:

- GPG signature verification
- Checksum verification (SHA-256)
- Pinned version hashes

### Future improvements

Recommended additions:

```bash
# Example: checksum verification for Go
local ver="1.23.4"
local expected_sha="a4da5273c96e898bc0e4b85767a3b3b2a10f0e1b9c0d1e2f3a4b5c6d7e8f9a0b"
curl -fsSL "https://go.dev/dl/go$ver.linux-amd64.tar.gz" -o /tmp/go.tar.gz
echo "$expected_sha  /tmp/go.tar.gz" | sha256sum -c || { setup_err "Checksum mismatch"; return 1; }
```

## Supply Chain Considerations

### Risk: Third-party install scripts

Multiple tools source from `curl | sh` patterns:

| Risk | Mitigation |
|------|------------|
| Compromised upstream URL | HTTPS provides transport security |
| Malicious installer script | None (trust-on-first-use) |
| Version mismatch | Pinned versions where possible (Go 1.23.4) |
| Script changed between runs | nvm/rustup/starship use stable URLs |

### Mitigation strategies

1. **Pin versions** — Go version is pinned (`ver="1.23.4"`). nvm, rustup, and
   Starship always fetch "latest" from their canonical URLs.
2. **HTTPS only** — Every download URL uses HTTPS. No HTTP fallback.
3. **`--offline` mode** — For air-gapped environments, `--offline` skips all
   downloads. Tools must be pre-installed.

## Secrets Handling

**The script does not handle secrets.** Specific design decisions:

| Secret type | Handling |
|-------------|----------|
| Git credentials | Passed via `--git-name`/`--git-email` or `.env`; written to `~/.gitconfig` by Git itself |
| SSH keys | Generated by `ssh-keygen`; stored in `~/.ssh/` by default |
| API tokens | Not handled |
| Passwords | Not handled (except sudo, which uses system `sudo -v`) |
| GPG keys | Not handled |

The script produces an append-only audit trail:

| Artifact | Contains | Retention |
|----------|----------|-----------|
| `install-*.log` | Every command executed, timestamps | Manual cleanup |
| `history.log` | Summary of each run (date, items) | Append-only |
| `backups/*/` | Pre-overwrite copies of user files | Manual cleanup |

## Backup Security

Backups preserve original file permissions:

```bash
cp -rL "$target" "$bak" 2>/dev/null
```

If a dotfile contains sensitive content (e.g., `~/.ssh/config`), the backup
copies it with its original permissions (usually `600`). The backup directory
itself is created with default `umask` permissions.

> [!NOTE]
> Backups of `~/.ssh/*` files are stored unencrypted in
> `~/.config/dotfiles-setup/backups/`. Ensure your backup directory has
> appropriate permissions: `chmod 700 ~/.config/dotfiles-setup/backups/`.

## Malicious Repo Detection

The script trusts the repository it's cloned into. To verify integrity:

```bash
git verify-commit HEAD     # If commits are signed
git log --show-signature   # Check commit signatures
```

There is **no** built-in integrity check for the repo itself.

## Safe Defaults

| Safety measure | Implementation |
|---------------|----------------|
| No `rm -rf /` | All `rm` calls specify exact paths, never `/` |
| Backup before overwrite | `backup_dotfile()` creates timestamped backups |
| Idempotent symlinks | `install_dotfile()` checks target before linking |
| Non-destructive TUI | Checklist returns empty if cancelled; no changes made |
| Offline-safe | `[ "$OFFLINE" = 1 ]` guards every network operation |
| Read-only dry run | `DRY_RUN=true` logs intent without executing |
